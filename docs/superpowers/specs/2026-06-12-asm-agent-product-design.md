# ASM 汇编工程生成智能体产品设计规格

**日期:** 2026-06-12

**目标平台:** Windows 桌面端优先

**产品形态:** 本地桌面应用 + 内置公司芯片 ASM 规范库 + 对话式 ASM 工程生成智能体

**参考风格:** 腾讯 Marvis 的现代桌面 AI 助手体验方向。仅提炼通用视觉与交互特征，不复制其品牌素材、图标、商标、专有文案或不可授权资源。

---

## 1. 产品一句话定义

ASM Agent 是一个面向公司芯片平台的本地 ASM 汇编工程生成智能体。用户用自然语言描述目标功能，智能体基于内置的公司芯片指令集、寄存器设计、芯片规格和 ASM 工程语法规则，生成完整、结构化、可审阅、可自检的 ASM 汇编代码工程。

产品核心不是编译器、仿真器、烧录器或通用 IDE，而是一个“会读公司芯片规范、会按公司 ASM 规则生成工程、会解释生成依据”的 ASM 工程智能体。

---

## 2. 背景与目标

公司芯片平台存在固定的 ASM 指令集、寄存器设计和工程语法要求。固件工程师或应用支持人员在开发示例工程、初始化代码、外设配置代码和中断处理框架时，需要严格遵循这些规范，不能使用未定义指令、非法寄存器、错误位域或不符合公司语法的 ASM 写法。

本产品目标是降低 ASM 工程生成门槛，使用户可以通过自然语言表达需求，例如“生成一个 Timer0 周期中断翻转 PA0 的完整 ASM 工程”，由 ASM Agent 自动完成需求理解、规范约束匹配、工程生成、代码解释和静态自检。

第一版重点解决以下问题：

- 用户不需要手动拼 ASM 工程骨架。
- 生成代码必须严格受公司芯片规范约束。
- 生成结果必须是完整工程，而不是孤立代码片段。
- 智能体必须能在资料不足或需求不明确时追问。
- 桌面 UI 需要呈现现代 AI 助手体验，参考 Marvis 的轻量、柔和、本地、智能感风格。

## 6. P0 硬性约束

ASM Agent 生成的所有 ASM 代码必须严格遵循公司芯片的指令集和寄存器设计要求。

### 6.1 指令约束

- 只能使用内置指令集中存在的 mnemonic。
- 只能使用内置指令集中存在的 `asm_syntax` 写法。
- 只能使用内置指令集中支持的操作数形式。
- 不得创造新的指令名、变体指令或未登记伪指令。
- 指令周期、影响标志位、语义说明必须以规范库为准。

### 6.2 操作数约束

- `k8` 必须限制在 8 位立即数范围。
- `k10` 必须限制在 10 位地址或立即数范围。
- `r8` 必须映射到合法的寄存器地址或公司规定的 8 位寄存器引用形式。
- `r8,b` 中的 bit 编号必须在 0 到 7 范围内，并且对应寄存器/位域必须存在。
- 空操作数指令不得追加操作数。

### 6.3 寄存器与位域约束

- 寄存器名称、地址、类型、复位值必须来自内置寄存器表。
- 位域名称、位宽、bit 位置和取值说明必须来自内置寄存器表或规格书。
- 不得虚构寄存器、位域、地址或访问方式。
- 对 `SFR`、`OPTION` 等不同寄存器类别，需要保留访问限制和配置前提。
- 对读写行为、清零行为、只读/只写/读清等访问语义，如果资料中存在说明，生成代码必须遵守。

### 6.4 向量与流程约束

- 复位入口必须遵循规格书中的复位向量要求。
- 中断入口必须遵循规格书中的中断向量要求。
- 如果用户需求涉及中断，但需求未说明中断源、使能条件或标志清除方式，智能体必须追问或生成带待确认项的计划。
- 如果用户需求涉及低功耗、WDT、Timer、GPIO 等外设，必须引用对应寄存器和位域依据。

### 6.5 ASM 工程语法约束

公司固定 ASM 语法必须作为内置规范的一部分维护。标签、注释、常量定义、文件包含、段定义、向量定义、伪指令、宏语法等必须来自公司 ASM 语法规则。

如果某个语法元素没有出现在内置规则中，智能体不得把它当作可用语法直接生成。需要输出待确认项或使用已登记的替代写法。

## 7. ASM 智能体行为规范

### 7.1 主行为流程

1. 接收用户自然语言需求。
2. 识别芯片平台和目标功能。
3. 加载对应内置规范库。
4. 判断需求是否包含足够信息。
5. 缺少关键信息时追问。
6. 信息足够时生成实现计划。
7. 用户确认计划后生成完整 ASM 工程。
8. 对生成结果执行规范自检。
9. 展示工程文件树、代码预览、规范符合性说明和风险提示。
10. 支持用户通过自然语言继续修改工程。

### 7.2 缺失信息处理

智能体不得为了完成生成而臆造芯片细节。出现以下情况时必须追问或标记待确认：

- 用户需求没有说明目标芯片平台，且项目没有默认芯片。
- 用户需求涉及外设，但无法从内置规范库确认寄存器或位域。
- 用户需求涉及中断，但没有明确中断源或清标志方式。
- 用户需求涉及定时周期，但缺少时钟源、分频、计数目标或时间单位。
- 用户需求涉及 GPIO，但缺少端口、方向、上下拉、开漏或初始电平信息。
- 用户需求涉及低功耗，但缺少唤醒源或进入/退出条件。
- 用户要求使用不在内置指令集中的指令或语法。

### 7.3 生成前计划

生成 ASM 工程前，智能体必须先输出简洁计划，包含：

- 目标功能理解
- 将生成的工程文件
- 涉及的外设模块
- 涉及的关键寄存器和位域
- 入口流程
- 是否使用中断
- 是否使用 WDT、Timer、GPIO、低功耗等特性
- 待确认项
- 规范风险

用户确认计划后再进入完整工程生成。

### 7.4 生成后解释

生成完成后，智能体必须解释：

- 每个文件的用途
- 关键 ASM 流程
- 使用了哪些指令形式
- 使用了哪些寄存器和位域
- 每段代码对应用户需求的哪一部分
- 哪些内容来自内置规范库
- 哪些内容仍需人工确认

## 8. ASM 工程交付物规范

### 8.1 工程文件树

每次生成完整 ASM 工程时，默认输出以下结构：

```text

    interrupt.asm
  src/
    main.asm
    gpio.asm
    timer.asm
  include/
    registers.inc
    bitfields.inc
    project_config.inc
  docs/
    requirements.md

    self-check-report.md
  tools/
    asm_static_check.json
  .vscode/
    tasks.json
    settings.json
    extensions.json
  README.md
  .gitignore
```

实际文件可根据需求裁剪，但必须保持工程结构清晰、命名稳定、职责明确。

### 8.2 启动代码

`startup/reset.asm` 应包含：

- 复位入口
- 初始化跳转或启动流程
- 必要的系统初始化占位
- 主程序入口跳转
- 与公司 ASM 语法匹配的标签和向量写法

如果内置语法规则中没有某类伪指令或段定义，智能体必须使用公司已登记写法，不得默认输出通用汇编器语法。

### 8.3 中断代码

`startup/interrupt.asm` 应包含：

- 中断入口
- 中断源判断框架
- 中断标志清除逻辑
- 返回中断指令

只有当需求涉及中断时才生成完整 ISR 逻辑。若中断需求不完整，应生成待确认项并追问。

### 8.4 主逻辑代码

`src/main.asm` 应包含：

- 主入口
- 外设初始化调用
- 主循环
- WDT 清除或低功耗逻辑，如果需求或规范要求存在

主循环必须与用户需求一致，不得加入用户未要求且规范未要求的功能。

### 8.5 外设模块代码

`src/gpio.asm`、`src/timer.asm` 等模块应按功能拆分。每个模块应包含：

- 初始化入口
- 关键寄存器设置
- 必要注释
- 对应规范依据

### 8.6 头文件和常量定义

`include/registers.inc` 和 `include/bitfields.inc` 应由内置寄存器规范生成或引用，不允许手写虚构寄存器。

常量命名应稳定、可读，并与公司已有命名习惯兼容。

### 8.7 文档交付物

`docs/requirements.md` 记录用户原始需求和智能体理解。

`docs/generation-plan.md` 记录生成前计划。

`docs/spec-compliance.md` 记录生成代码使用的关键指令、寄存器、位域和规范依据。

`docs/self-check-report.md` 记录自检结果、待确认项、阻断问题和非目标说明。

### 8.8 项目配置

`.gitignore` 至少排除：

*.o
*.obj
*.elf
*.bin
*.hex
*.map
*.lst

`.vscode/` 示例配置只包含查看、静态检查、打开报告、导出工程等本地辅助任务。第一版不得默认包含编译、仿真或烧录任务。

## 9. 自检与规范符合性

自检是生成质量保障，不是产品主定位。用户感知上，ASM Agent 仍是自然语言生成工程的智能体。

### 9.1 自检范围

自检必须覆盖：

- 指令 mnemonic 是否存在
- 指令写法是否匹配内置 `asm_syntax`
- 操作数数量是否正确
- 操作数类型是否正确
- 立即数范围是否正确
- 寄存器名称和地址是否存在
- 位域名称和 bit 位置是否存在
- 跳转和调用目标是否在规格允许范围内
- 复位入口是否存在
- 中断入口是否按需求和规范生成
- 工程文件是否完整
- 文档是否记录生成依据
- 指令编码是否满足 `opcode_pattern`、`mask_hex`、`value_hex` 的一致性

### 9.2 自检结果等级

自检结果分为：

- 通过: 可导出工程。
- 警告: 可导出，但必须提示用户阅读风险。
- 待确认: 缺少需求细节或内部规范未完全覆盖，需要用户确认。
- 阻断: 存在非法指令、非法寄存器、非法位域、越界操作数或不符合内置语法的 ASM，不允许默认导出。

### 9.3 编码级一致性

对于每条可编码指令，系统应根据内置指令表检查：

- `word_bits` 是否匹配
- `opcode_pattern` 是否能覆盖固定 bit 和可变操作数 bit
- 编码结果按 `mask_hex` 与 `value_hex` 检查是否一致
- 操作数填入后是否超出可编码范围

该能力用于发现生成错误，不表示产品会生成烧录产物。

## 10. 桌面 UI/UX 设计

### 10.1 风格方向

UI 参考 Marvis 的现代桌面 AI 助手体验方向，强调：

- AI 助手感
- 本地模式可信感
- 清晰的信息层级
- 柔和但有识别度的科技感
- 面向开发者的专业代码阅读体验
- 流畅但克制的状态反馈

不复制 Marvis 或腾讯的品牌素材、图标、商标、专有文案或特定品牌色值。

### 10.2 主界面信息架构

主界面采用三栏结构：

- 左侧: 芯片平台、内置规范版本、任务历史、工程草稿、导出记录。
- 中央: 自然语言输入、智能体对话、需求理解、生成计划、生成按钮。
- 右侧: ASM 工程文件树、代码预览、规范符合性摘要、风险提示。

用户核心视线应落在中央对话区，工程预览和自检信息作为辅助。

### 10.3 普通用户主流程

普通用户主流程不包含“导入规范”：

1. 打开应用。
2. 选择或确认芯片平台。
3. 输入自然语言需求。
4. 查看智能体生成计划。
5. 确认计划。
6. 生成 ASM 工程。
7. 查看代码、工程文件树和规范符合性报告。
8. 导出工程。
9. 继续通过对话修改工程。

### 10.4 关键界面

#### 主界面

展示当前芯片平台、规范版本、对话区、生成计划、工程预览和自检摘要。

#### 项目创建界面

包含项目名称、芯片平台、保存位置、工程模板类型和初始需求输入。

#### 芯片平台与规范版本界面

展示内置芯片规范版本、更新时间、适用芯片型号和变更摘要。该界面用于查看，不作为普通用户导入规范的入口。

#### 功能需求输入界面

以多行自然语言输入为主，支持输入外设、目标行为、初始化要求、中断要求、低功耗要求和输出工程名称。

#### ASM 代码生成结果界面

展示生成计划、文件树、代码预览、规范符合性摘要和导出按钮。

#### 工程文件预览界面

以文件树和代码预览为主，支持查看 `startup/`、`src/`、`include/`、`docs/` 等目录。

#### 自检与规范符合性界面

展示通过项、警告项、待确认项和阻断项。每项应包含文件、行号、问题说明、规范依据和建议处理方式。

#### 错误提示与修复建议界面

以诊断卡片形式展示，例如非法指令、未知寄存器、位域不存在、立即数越界、需求信息不足等。

#### 设置界面

普通设置包括主题、字体、代码字号、默认导出路径、历史记录保留策略、本地日志策略。

管理员或内部维护设置可包含规范库版本查看、规范库更新包安装和规则库诊断。

### 10.5 视觉规则

- 背景: 浅色中性背景为主，代码区使用深色背景。
- 强调色: 使用低饱和蓝、青蓝、蓝紫作为强调色。
- 字体: Windows 系统 UI 字体优先，例如 `Segoe UI`、`Microsoft YaHei UI`。
- 代码字体: `Cascadia Mono`、`Consolas` 或等宽字体。
- 圆角: 按钮、输入框、卡片统一中等圆角。
- 阴影: 使用轻量阴影区分窗口、浮层和主面板。
- 动效: 对话生成、计划展开、工程生成、自检状态变化使用轻量过渡。
- 图标: 使用通用线性图标，不使用腾讯或 Marvis 专属图标。

### 10.6 交互规则

- 用户输入需求后，优先展示智能体理解和生成计划，而不是立即大量输出代码。
- 用户确认计划后再生成完整工程。
- 若需求不足，智能体以对话形式追问，不弹出复杂配置表单。
- 修改需求时，智能体说明哪些文件会被重写，哪些文件保持不变。
- 导出前展示自检状态。
- 阻断错误未解决时，默认不允许导出。

## 11. 安装、更新与卸载

### 11.1 安装流程

Windows 安装向导包含：

1. 欢迎页: 说明这是本地 ASM 工程智能体。
2. 安装位置页: 默认安装到公司标准应用目录，允许用户修改。
3. 组件页: 主程序、内置芯片规范库、示例工程、离线帮助文档。
4. 安装进度页: 显示当前步骤和进度。
5. 完成页: 提供启动应用和打开示例工程选项。

安装过程视觉风格与主应用保持一致，强调本地、安全、离线可用。

### 11.2 更新流程

更新分为：

- 应用程序更新
- 内置规范库更新
- 示例工程更新
- 离线帮助文档更新

更新前必须展示版本差异和变更摘要。规范库更新必须记录新旧版本，以便追溯已有工程使用的生成依据。

### 11.3 卸载流程

卸载时：

- 默认保留用户导出的 ASM 工程。
- 可选删除本地缓存、历史对话和日志。
- 明确提示不会删除用户手动保存到其他目录的工程。

## 12. 验收标准

第一版产品验收应满足：

1. 用户可以通过自然语言描述 ASM 工程需求。
2. 系统能基于内置公司芯片规范生成实现计划。
3. 用户确认计划后，系统能生成完整 ASM 工程目录。
4. 生成代码只能使用内置指令集、寄存器表、位域和公司 ASM 语法。
5. 如果需求超出规范或信息不足，智能体必须追问或标记待确认。
6. 生成结果包含工程文件树、核心 ASM 文件、寄存器/位域定义、README、文档和基础项目配置。
7. 系统能输出规范符合性说明和自检报告。
8. 系统能识别非法指令、未知寄存器、未知位域、操作数越界和不符合内置语法的问题。
9. UI 主流程不暴露普通用户导入规范按钮。
10. UI 以对话式 ASM 智能体为中心，而不是以配置面板或校验工具为中心。
11. 产品不声称完成编译、仿真、烧录或真实硬件验证。
12. 安装、更新、卸载流程保持与主应用一致的桌面体验。

## 13. 示例用户旅程

### 13.1 生成 Timer0 中断工程

用户输入：

```text
生成一个 Timer0 周期中断翻转 PA0 的完整 ASM 工程，需要启动入口、主循环、中断处理和注释。
```

智能体响应：

1. 确认当前芯片平台为 HK64S8x。
2. 自动加载内置规范库。
3. 检查 Timer0、PA0、中断入口、相关寄存器和指令形式。
4. 如果定时周期缺失，追问目标周期、时钟源或分频要求。
5. 输出生成计划。
6. 用户确认后生成工程。
7. 展示文件树、代码预览和规范符合性说明。

### 13.2 修改需求

用户输入：

```text
把 PA0 改成 PA3，并且主循环里加上 WDT 清零。
```

智能体响应：

1. 判断该修改影响 GPIO 模块、主循环和文档。
2. 检查 PA3 相关寄存器/位域是否存在。
3. 检查 WDT 清零指令是否存在于内置指令集。
4. 重新生成受影响文件。
5. 更新自检报告。

## 14. 风险与待确认项

### 14.1 需要内部确认

- 公司固定 ASM 语法的完整定义，包括标签、注释、伪指令、段定义、常量定义和文件包含语法。
- 工程模板的标准目录命名。
- 寄存器命名是否需要统一替换 `-`、`_` 等符号。
- OPTION 寄存器和 SFR 寄存器在生成代码中的访问限制。
- 中断源优先级、标志清除方式和中断使能顺序。
- Timer 周期计算是否进入第一版，或仅生成结构化初始化框架。
- 是否需要管理员模式维护内置规范库。

### 14.2 主要产品风险

- 用户自然语言需求过于模糊，导致生成计划需要多轮追问。
- 规格书、指令表和寄存器表之间可能存在命名差异。
- 公司 ASM 语法如果未完整内置，工程输出可能只能生成草案。
- 生成代码即使通过静态自检，也不代表真实硬件行为已验证。

---

## 15. 结论

ASM Agent 第一版应聚焦为“内置公司芯片 ASM 规范的本地自然语言工程生成智能体”。用户只需要描述功能需求，智能体负责理解、计划、生成、解释和自检。

产品 UI 参考 Marvis 的现代桌面 AI 助手体验方向，但功能主轴始终是 ASM 汇编工程生成。规范库内置、代码受约束、工程可审阅、自检可追溯，是第一版产品的核心价值。
